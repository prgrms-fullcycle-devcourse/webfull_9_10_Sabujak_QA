# qa-playwright

사부작 타임캡슐 서비스 전용 E2E QA 스킬이다. 이 레포는 프론트엔드와 백엔드 구현 레포와 분리되어 있으며, Codex가 Playwright MCP를 통해 실제 사용자 흐름을 검증할 때 사용한다.

## 목적

- `../webfull_9_10_Sabujak_FE`와 `../webfull_9_10_Sabujak_BE`를 직접 수정하지 않고도 QA 시나리오를 반복 실행한다.
- 브라우저 기준 사용자 플로우와 API 응답 규칙을 함께 검증한다.
- mock 환경과 real 환경의 차이를 분리해서 기록하고 회귀 테스트에 반영한다.

## 기본 원칙

- 테스트 시작 전에 대상 환경을 명확히 구분한다: `mock`, `staging`, `real`.
- UI 동작만 보지 말고 네트워크 요청과 응답 페이로드를 함께 확인한다.
- 슬러그, 공개 시점, 만료 시점, 관리자 비밀번호 같은 도메인 규칙은 API 결과까지 검증한다.
- 실패 시 재현 경로, 입력값, 기대값, 실제값을 함께 남긴다.

## 사부작 핵심 도메인 체크포인트

### 1. slug 선점과 reservationToken 기반 capsule 생성

- capsule 생성은 선행 slug 선점 단계와 본 생성 단계로 분리되어야 한다.
- slug 선점 성공 후 `reservationToken`이 발급되는지 확인한다.
- capsule 생성 요청이 해당 `reservationToken` 없이 실패하는지 확인한다.
- 이미 선점되었거나 사용 완료된 slug를 재사용할 때 차단되는지 확인한다.
- mock 환경에서는 토큰 검증이 느슨할 수 있으므로 real 환경에서 반드시 재검증한다.

### 2. 공개 전/후 capsule 조회 분기

- `openAt` 이전에는 공개 조회에서 capsule 본문이 차단되거나 제한된 응답을 주는지 확인한다.
- `openAt` 이후에는 동일 slug로 공개 조회가 정상 전환되는지 확인한다.
- 비공개 상태에서도 작성자용 또는 관리자용 조회 경로가 있다면 권한별 응답 차이를 확인한다.
- mock 환경에서 시간을 고정하거나 즉시 공개 처리하는 경우가 있으므로, real 환경에서는 실제 시간 기준 분기를 다시 검증한다.

### 3. message 작성

- capsule 생성 또는 수정 흐름에서 message가 저장되고 조회에 반영되는지 확인한다.
- 빈 문자열, 최대 길이 경계값, 줄바꿈 포함 메시지 처리를 확인한다.
- 공개 전에는 message 노출 범위가 제한되는지, 공개 후에는 정상 노출되는지 확인한다.

### 4. duplicate nickname 차단

- 동일 capsule 또는 동일 메시지 작성 컨텍스트에서 중복 nickname 사용이 차단되는지 확인한다.
- 중복 차단 시 사용자에게 이해 가능한 오류 메시지가 노출되는지 확인한다.
- mock 환경에서 중복 검사가 생략되거나 메모리 상태에 의존할 수 있으므로 real 환경에서 병행 검증한다.

### 5. capsule 수정/삭제 시 관리자 비밀번호 검증

- 수정과 삭제 요청 모두 관리자 비밀번호(`password`) 검증을 거치는지 확인한다.
- 잘못된 비밀번호로 수정/삭제가 차단되는지 확인한다.
- 올바른 비밀번호에서는 수정/삭제가 수행되고 후속 조회 결과가 일치하는지 확인한다.
- UI 에러 메시지와 API 에러 코드가 일관적인지 함께 확인한다.

### 6. openAt 변경 시 expiresAt 재계산

- capsule 수정 시 `openAt`을 바꾸면 `expiresAt`도 도메인 규칙에 맞게 함께 재계산되는지 확인한다.
- FE 화면 표시값과 API 응답값이 동일한지 비교한다.
- mock 환경에서 `expiresAt`이 고정값일 수 있으므로, real 환경에서는 실제 계산식이 반영되는지 우선 확인한다.

### 7. mock 환경과 real 환경 차이

- mock은 UI 흐름 검증과 빠른 회귀 확인에 사용한다.
- real은 인증, 중복 검증, 토큰 유효성, 시간 기반 공개 분기 같은 실제 서버 규칙 검증에 사용한다.
- 같은 테스트라도 mock 통과만으로 종료하지 말고, 실서버 규칙이 있는 시나리오는 real 결과를 별도로 기록한다.
- mock에서 성공하고 real에서 실패하면 API 계약 불일치 또는 프론트 가정 오류 가능성을 우선 의심한다.

## Codex 작업 방식

1. 환경 변수에서 `APP_BASE_URL`, `API_BASE_URL`, `TEST_CAPSULE_SLUG`, `TEST_ADMIN_PASSWORD` 값을 읽는다.
2. `QA_ENV` 값을 보고 대상이 `mock`, `staging`, `real` 중 무엇인지 먼저 확인하고 테스트 범위를 선언한다.
3. Playwright MCP로 화면을 조작하면서 네트워크 요청을 관찰한다.
4. 필요하면 capsule 생성 전 slug 선점 요청과 응답 토큰을 직접 추적한다.
5. 실패 케이스와 성공 케이스를 모두 재현한다.
6. 결과는 `재현 절차 / 기대 결과 / 실제 결과 / 환경(mock|real)` 형식으로 요약한다.

## 자동 QA와의 연결

- 이 레포의 GitHub Actions와 Playwright 테스트는 외부로 떠 있는 환경을 대상으로 실행한다.
- 로컬에서는 인접 FE/BE 레포를 사람이 직접 실행한 뒤 같은 테스트를 재사용한다.
- CI에서는 `APP_BASE_URL`, `API_BASE_URL`, `TEST_CAPSULE_SLUG`, `TEST_ADMIN_PASSWORD`가 준비된 경우에만 QA를 실행한다.
- mock 환경에서는 현재 구현상 느슨한 동작이 있으므로, duplicate nickname 차단과 관리자 비밀번호 실패는 "미구현 차이"로 기록한다.
- real 또는 staging 환경에서는 예약 토큰, 공개 전/후 분기, duplicate nickname 차단, 관리자 비밀번호 실패, `openAt` 변경 시 `expiresAt` 재계산까지 실제 규칙으로 검증한다.

## 자주 수행할 QA 요청 예시

- mock 환경에서 capsule 생성 전체 플로우를 검증하고 `reservationToken` 사용 여부까지 확인해라.
- real 환경에서 공개 전 capsule 조회와 공개 후 조회 응답 차이를 검증해라.
- duplicate nickname 차단이 UI와 API에서 일치하는지 확인해라.
- 관리자 비밀번호가 틀릴 때 capsule 삭제가 차단되는지 검증해라.
- `openAt` 수정 후 `expiresAt`이 재계산되는지 API 응답까지 포함해 확인해라.
