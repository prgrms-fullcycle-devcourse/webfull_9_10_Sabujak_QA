# 백엔드 `main` QA 후속 대응 계획

- 작성일: 2026-03-25
- 기준 보고서: `docs/reports/backend/2026-03-25-backend-main-qa-incident-report.md`
- 반영 사항:
  - `POST /capsules/:slug/verify`는 현재 **의도된 mock 상태의 미구현 API**로 간주한다.
  - 헬스체크의 기준 경로는 `/healthCheck`로 확정한다.

## 1. 계획의 기준점

이번 후속 계획은 QA 보고서의 항목을 그대로 "전부 즉시 수정"하는 방식이 아니라, 현재 제품 상태와 구현 정책을 반영해 우선순위를 다시 정리하는 데 목적이 있다.

핵심 기준은 아래와 같다.

1. `verify`는 현재 버그 수정 대상이 아니라 **미구현 기능 관리 대상**이다.
2. 헬스체크는 `/health`로 바꾸는 것이 아니라, 문서와 테스트를 `/healthCheck` 기준으로 통일한다.
3. 즉시 수정이 필요한 실제 런타임 장애와, 아직 공개 범위를 제한해야 하는 미구현 기능을 구분해서 대응한다.

## 2. 이슈 재분류

### A. `verify` 이슈: 인증 버그가 아니라 미구현 기능 노출 관리 이슈

- 현재 상태:
  - `POST /capsules/:slug/verify`는 repository에서 mock 응답을 반환한다.
  - 이 상태는 현재 의도된 미구현 상태로 본다.
- 해석:
  - 지금 시점에서 이 API를 "정상 인증 API"로 보고 즉시 구현을 요구하는 것은 범위를 벗어난다.
  - 대신, **미구현 API가 완성된 기능처럼 오해되지 않도록 계약과 운영 범위를 관리하는 것**이 우선이다.
- 핵심 리스크:
  - FE/QA가 이 응답을 실제 검증 결과로 신뢰하면 잘못된 사용자 흐름을 설계할 수 있다.
  - 문서에 완성 기능처럼 남아 있으면 이후 회귀 때마다 같은 혼선이 반복된다.

### B. duplicate nickname 이슈: 즉시 수정이 필요한 실제 런타임 장애

- 현재 상태:
  - 메시지 중복 닉네임 충돌이 명세상 `409`여야 하지만 실제로는 `500`이 발생한다.
- 해석:
  - 이 항목은 여전히 즉시 수정 대상이다.
  - 사용자의 정상적인 충돌 상황을 서버 장애처럼 노출하고 있으므로 계약 위반이자 운영 관점의 잡음이다.

### C. healthCheck 이슈: 라우트 변경이 아니라 문서 기준 정리 이슈

- 현재 상태:
  - 실제 구현은 `/healthCheck`다.
  - 일부 문서/QA 시나리오가 `/health`를 참조하고 있다.
- 해석:
  - 이 이슈의 대응 방향은 라우트를 `/health`로 바꾸는 것이 아니다.
  - **문서, 테스트, 운영 체크 기준을 `/healthCheck`로 통일**하는 것이 맞다.

### D. 공통 구조 이슈: 문서-테스트-런타임 기준점 불일치

- 현재 상태:
  - 문서와 실제 구현이 어긋난 항목이 있고, 테스트는 이를 충분히 막지 못한다.
- 해석:
  - 이번 대응의 핵심은 "무엇을 구현할지"만이 아니라, "미구현 기능과 실제 제공 기능을 어떻게 명확히 구분할지"까지 포함해야 한다.

## 3. 우선순위별 대응 계획

### 3.1 지금 바로 반영할 계획

1. duplicate nickname의 `500 -> 409` 변환 문제를 수정한다.
   - `error.code`뿐 아니라 `error.cause?.code`를 해석한다.
   - 가능하면 `constraint` 이름까지 확인해 정확히 닉네임 충돌만 `DuplicateNicknameException`으로 변환한다.
   - slug unique 충돌 처리도 같은 방식으로 보강한다.

2. `verify`를 미구현 API로 명시한다.
   - `docs/API_SPEC.md`, OpenAPI 설명, QA 기준 문서에서 해당 API를 "mock/미구현" 상태로 표기한다.
   - FE와 QA가 이 API를 실제 인증 보장 수단으로 사용하지 않도록 범위를 명확히 적는다.
   - 이번 스프린트에서 구현 대상이 아니라는 점을 문서상으로 분명히 남긴다.

3. 헬스체크 기준 경로를 `/healthCheck`로 통일한다.
   - `README`
   - `docs/API_SPEC.md`
   - QA smoke test 문서
   - 필요 시 OpenAPI description

### 3.2 이번 스프린트 내 계획

1. HTTP 계약 회귀 테스트를 추가한다.
   - duplicate nickname -> `409`
   - `/healthCheck` -> `200`
   - 문서상 공개된 핵심 path가 실제 라우트와 일치하는지 확인

2. `verify`의 임시 운영 원칙을 정한다.
   - FE에서 선검증 API로 의존하지 않는다.
   - 수정/삭제의 실제 권한 판단은 기존 `PATCH`/`DELETE`의 비밀번호 검증 결과를 기준으로 본다.
   - QA 체크리스트에서 `verify`는 기능 완료 항목이 아니라 "미구현 mock 확인 항목"으로 분리한다.

3. 문서 드리프트를 줄이는 기준점을 만든다.
   - 라우트 변경 시 `README`, `API_SPEC`, OpenAPI를 함께 갱신한다.
   - 이번에는 `/healthCheck`를 기준값으로 삼는다.

### 3.3 구현 시점 이후를 위한 준비 계획

1. `verify` 구현 전 선행 정리를 한다.
   - expected contract 확정
   - `404 / 403 / 200` 응답 규격 확정
   - FE 사용 시나리오 정의
   - route/integration test 시나리오 초안 작성

2. DB 예외 해석 로직을 공통화한다.
   - repository별 ad-hoc 처리 대신 helper로 표준화한다.
   - 새 unique 제약이 생겨도 오탐 없이 매핑되도록 만든다.

3. 미구현 기능 관리 규칙을 추가한다.
   - mock 상태의 API는 문서에서 명확히 표시한다.
   - QA 완료 범위와 백로그 범위를 구분한다.
   - production path에 남아 있는 mock 코드가 "완성 기능"으로 오해되지 않도록 체크리스트를 둔다.

## 4. 권장 실행 순서

1. duplicate nickname 예외 변환 수정
2. `/healthCheck` 기준으로 문서 정리
3. `verify`를 미구현 mock API로 문서에 명시
4. HTTP 계약 회귀 테스트 추가
5. `verify` 정식 구현을 위한 계약 초안 작성

이 순서가 좋은 이유는, 먼저 실제 사용자 영향이 있는 런타임 장애를 닫고, 그다음 미구현 기능과 실제 제공 기능의 경계를 문서와 테스트에서 명확히 해야 혼선이 줄어들기 때문이다.

## 5. 작업 항목 제안

- 백엔드 즉시 작업:
  - duplicate nickname `409` 복구
  - slug unique 예외 변환 보강
  - `/healthCheck` 기준 문서 수정
- 백엔드 문서 작업:
  - `verify`를 mock/미구현 상태로 명시
  - health endpoint 설명을 `/healthCheck`로 통일
- FE/QA 공조 작업:
  - `verify` 의존 제거 또는 보류
  - `/healthCheck` 기준 smoke test 갱신
  - duplicate nickname 응답 코드 재검증

## 6. 최종 판단

이번 계획에서 가장 중요한 건 두 가지다.

1. 진짜 장애는 즉시 고친다.
2. 아직 미구현인 기능은 미구현이라고 명확히 관리한다.

따라서 현재 기준에서 우선순위 1번은 duplicate nickname의 `500` 문제이고, `verify`는 즉시 구현 대상이 아니라 문서와 QA 범위를 정리해야 하는 관리 항목이다. 헬스체크는 `/healthCheck`를 정답으로 두고 관련 문서와 테스트를 그 기준으로 맞추는 방향이 맞다.
